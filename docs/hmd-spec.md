# Historica Markdown (HMD) Format Specification

Version: 1.0.0-draft

## 1. Overview

Historica Markdown (HMD) is the storage format for Obsidian Historica timeline
data. It is a strict subset of standard markdown, designed so that:

1. Any HMD file is valid markdown, renderable by any markdown viewer.
2. Obsidian renders it as a readable document without the Historica plugin.
3. The Historica plugin can parse it losslessly into structured timeline data
   and serialize it back without content drift.

### 1.1 Design Goals

- **Human-first**: a user should be able to read and edit the file by hand.
- **Round-trip safe**: parse then serialize must produce byte-identical output.
- **Obsidian-native**: uses YAML frontmatter, inline fields (`key:: value`),
  and Obsidian comments (`%%...%%`) — no invented syntax.
- **Progressive enhancement**: without the plugin, the file is a readable
  markdown document with dates and paragraphs. With the plugin, it renders
  as an interactive timeline.

### 1.2 File Location

HMD files are stored in the vault at `historica-data/{blockId}.md`, replacing
the previous `historica-data/{blockId}.json` format.

---

## 2. Document Structure

An HMD file has two parts:

```
┌─────────────────────┐
│  YAML Frontmatter   │  ← timeline settings
├─────────────────────┤
│  Entry 1            │
│─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│  ---                │  ← horizontal rule separator
│─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│  Entry 2            │
│─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│  ---                │
│─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│  Entry N            │
└─────────────────────┘
```

The frontmatter is **required**. Entries are separated by horizontal rules
(`---` on its own line, with blank lines before and after). The last entry
has no trailing separator.

---

## 3. Frontmatter

The YAML frontmatter block contains timeline-level settings.

### 3.1 Schema

| Field       | Type    | Required | Default   | Description                  |
|-------------|---------|----------|-----------|------------------------------|
| `blockId`   | string  | yes      | —         | Unique block identifier      |
| `autoSave`  | boolean | no       | `true`    | Auto-save on state change    |
| `header`    | string  | no       | `""`      | HTML header content          |
| `footer`    | string  | no       | `""`      | HTML footer content          |

### 3.2 Rules

1. The frontmatter block MUST be the first thing in the file.
2. It is delimited by `---` on the first line and `---` on a subsequent line.
3. Unknown fields MUST be preserved on round-trip (forward compatibility).
4. `blockId` MUST be present. A file without `blockId` is invalid HMD.

### Example 1: Minimal frontmatter

````
---
blockId: "1992306556"
---
````

Parses to:

```
settings.blockId  = "1992306556"
settings.autoSave = true          (default)
settings.header   = ""            (default)
settings.footer   = ""            (default)
```

### Example 2: Full frontmatter

````
---
blockId: "1992306556"
autoSave: false
header: "<b>World History</b>"
footer: "<i>Generated by Historica</i>"
---
````

Parses to:

```
settings.blockId  = "1992306556"
settings.autoSave = false
settings.header   = "<b>World History</b>"
settings.footer   = "<i>Generated by Historica</i>"
```

## 4. Entry

Each entry represents one timeline event (`PlotUnitNg`). An entry consists of
three parts, in order:

```
## {title}              ← heading (required)
{fields}                ← inline fields, one per line (at least date required)
                        ← blank line
{body}                  ← paragraph text (required)
```

### 4.1 Heading (Title)

An H2 heading (`## `). The text becomes `parsedResultText`.

**Rules:**
1. MUST be H2 level. H1 is reserved for future use. H3+ inside body text is
   allowed but not parsed as entry titles.
2. The heading text is trimmed of leading/trailing whitespace.

### 4.2 Inline Fields

Lines immediately after the heading, before the first blank line, using
Obsidian's inline field syntax: `key:: value`.

| Field          | Type   | Required | Maps to                     |
|----------------|--------|----------|-----------------------------|
| `date`         | string | yes      | `time.value` (ISO 8601 or unix timestamp) |
| `date-style`   | string | no       | `time.style` — `"unix"` or `"free"`. Default: inferred |
| `source`       | string | no       | `filePath` — vault-relative path to source note |

**Date inference rules:**
- If `date` value matches ISO 8601 (`YYYY-MM-DD` or `YYYY-MM-DDTHH:MM:SS`),
  it is stored as a unix timestamp with `style: "unix"`.
- If `date` value is a plain integer (possibly negative), it is treated as a
  raw unix timestamp in milliseconds with `style: "unix"`.
- Otherwise, `style: "free"` and the value is stored as-is.

### 4.3 Internal Fields (Obsidian Comments)

Internal state is stored as Obsidian comments (`%%...%%`) on their own lines,
after the inline fields and before the body paragraph. These are invisible in
Obsidian's reading mode.

| Comment field  | Type    | Default | Maps to        |
|----------------|---------|---------|----------------|
| `%%id: ...%%`  | string  | auto    | `id`           |
| `%%hidden%%`   | flag    | absent  | `isHidden: true` |

**Rules:**
1. `%%id: ...%%` — if absent, the parser generates a new ID on first read.
2. `%%hidden%%` — presence means `isHidden: true`. Absence means `false`.
3. `isExpanded` is NOT stored — it is UI-only state, always defaults to `false`.
4. `nodePos` is NOT stored — it is a parse artifact, recomputed on load.
5. `fileParent` is NOT stored — it is derived from `source` path.
6. Unknown `%%key: value%%` comments MUST be preserved on round-trip.

### 4.4 Body

Everything after the first blank line following the fields/comments, up to the
next `---` separator (or end of file), is the body text. This maps to
`sentence`.

**Rules:**
1. The body is trimmed of leading/trailing blank lines.
2. The body may contain multiple paragraphs, inline formatting, links, etc.
3. The body MUST NOT contain `---` on its own line (ambiguous with separator).
   If the original sentence contains `---`, it must be escaped as `\---`.

### Example 3: Basic entry

````
## July 14, 1789
date:: 1789-07-14
source:: flow-test.md

The French Revolution began on July 14, 1789.
````

Parses to:

```
parsedResultText = "July 14, 1789"
time.value       = "-5694926400000"    (unix ms for 1789-07-14)
time.style       = "unix"
filePath         = "flow-test.md"
sentence         = "The French Revolution began on July 14, 1789."
id               = (auto-generated)
isExpanded       = false
isHidden         = false
attachments      = []
```

### Example 4: Entry with internal state

````
## September 1, 1939
date:: 1939-09-01
source:: user-test-ww2.md
%%id: 2044819733%%
%%hidden%%

The war in Europe began when Germany invaded Poland on September 1, 1939.
Britain and France declared war on Germany two days later.
````

Parses to:

```
parsedResultText = "September 1, 1939"
time.value       = "-957268800000"
time.style       = "unix"
filePath         = "user-test-ww2.md"
sentence         = "The war in Europe began when Germany invaded Poland on September 1, 1939.\nBritain and France declared war on Germany two days later."
id               = "2044819733"
isHidden         = true
```

### Example 5: Entry with free-form date

````
## The Bronze Age
date:: circa 3300 BCE
date-style:: free

The Bronze Age began around 3300 BCE in the Near East.
````

Parses to:

```
parsedResultText = "The Bronze Age"
time.value       = "circa 3300 BCE"
time.style       = "free"
```

### Example 6: Entry with unix timestamp

````
## Apollo 11
date:: -14182940000
source:: space-history.md

Humans first walked on the Moon on July 20, 1969.
````

Parses to:

```
parsedResultText = "Apollo 11"
time.value       = "-14182940000"
time.style       = "unix"
```

## 5. Entry Separator

Entries are separated by a horizontal rule: a line containing only `---`,
with a blank line before and after it.

### 5.1 Rules

1. The separator is `---` on its own line (three hyphens, no leading spaces).
2. There MUST be a blank line before and after the separator.
3. The last entry in the file has NO trailing separator.
4. The frontmatter closing `---` is NOT an entry separator — the first entry
   starts after the frontmatter block, separated by a blank line.

### Example 7: Two entries with separator

````
---
blockId: "100"
---

## Event One
date:: 2000-01-01

The first event.

---

## Event Two
date:: 2000-12-31

The second event.
````

Parses to 2 entries: "Event One" and "Event Two".

### Example 8: Single entry (no separator)

````
---
blockId: "101"
---

## Only Event
date:: 2020-06-15

This is the only entry.
````

Parses to 1 entry.

## 6. Attachments

Attachments are listed as a sub-list under an `attachments:` marker line,
after the inline fields and before the body.

### 6.1 Syntax

```
attachments:
- [[path/to/image.png]]
- [[path/to/document.pdf]]
```

Each list item is an Obsidian wiki-link (`[[...]]`). The path inside the link
maps to `Attachment.path`. The `Attachment.id` is auto-generated from the path.

### 6.2 Rules

1. The `attachments:` line has no `::` — it is a plain label, not an inline
   field (to avoid Dataview interpreting the list as a field value).
2. Each attachment is a `- [[path]]` list item.
3. If there are no attachments, the entire block is omitted (not
   `attachments:` with an empty list).
4. Attachments appear after inline fields and comments, before the blank line
   that starts the body.

### Example 9: Entry with attachments

````
## D-Day
date:: 1944-06-06
source:: ww2-notes.md
%%id: 9988776655%%
attachments:
- [[images/d-day-beach.jpg]]
- [[documents/operation-overlord.pdf]]

On June 6, Allied forces landed on the beaches of Normandy in the largest
amphibious invasion in military history.
````

Parses to:

```
parsedResultText = "D-Day"
attachments      = [
  {id: (hash of "images/d-day-beach.jpg"), path: "images/d-day-beach.jpg"},
  {id: (hash of "documents/operation-overlord.pdf"), path: "documents/operation-overlord.pdf"},
]
sentence         = "On June 6, Allied forces landed on the beaches of Normandy in the largest\namphibious invasion in military history."
```

## 8. Edge Cases

### Example 11: Entry body with multiple paragraphs

````
## The Renaissance
date:: 1400-01-01

The Renaissance was a cultural movement that began in Italy.

It spread across Europe over the following centuries, fundamentally
changing art, science, and philosophy.
````

Parses to:

```
sentence = "The Renaissance was a cultural movement that began in Italy.\n\nIt spread across Europe over the following centuries, fundamentally\nchanging art, science, and philosophy."
```

The body preserves paragraph breaks (double newline).

### Example 12: Entry with no source field

````
## Some Event
date:: 2023-01-15

A manually added entry with no source file.
````

Parses to:

```
filePath = ""    (empty string, not undefined)
```

### Example 13: Entry with markdown formatting in body

````
## The Moon Landing
date:: 1969-07-20
source:: space.md

Neil Armstrong said: *"That's one small step for man, one giant leap
for mankind."* The **Apollo 11** mission was a success.
````

Parses to:

```
sentence = "Neil Armstrong said: *\"That's one small step for man, one giant leap\nfor mankind.\"* The **Apollo 11** mission was a success."
```

The body preserves inline markdown formatting as-is.

### Example 14: Empty file (frontmatter only, no entries)

````
---
blockId: "999"
---
````

Parses to:

```
settings.blockId = "999"
units            = []
```

### Example 15: Unknown frontmatter fields preserved

````
---
blockId: "100"
customField: "hello"
---

## Test
date:: 2020-01-01

Body text.
````

On round-trip, `customField: "hello"` MUST be preserved in the output
frontmatter.

### Example 16: Unknown comment fields preserved

````
## Test
date:: 2020-01-01
%%id: abc%%
%%customTag: some value%%

Body text.
````

On round-trip, `%%customTag: some value%%` MUST appear in the output.

## 9. Serialization Order

When writing an HMD file from in-memory data, the serializer MUST produce
fields in this exact order within each entry:

1. `## {title}` heading
2. `date:: {value}` (always first field)
3. `date-style:: {value}` (only if style is `"free"`)
4. `source:: {value}` (only if non-empty)
5. `%%id: {value}%%`
6. `%%hidden%%` (only if true)
7. Any unknown preserved comments
8. `attachments:` block (only if non-empty)
9. Blank line
10. Body text

This ordering ensures deterministic output for round-trip fidelity.

## 10. Migration from JSON

Existing `historica-data/{blockId}.json` files are converted to
`historica-data/{blockId}.md` by:

1. Reading the JSON `HistoricaFileData`.
2. Serializing to HMD format per this spec.
3. Writing the `.md` file.
4. Deleting the `.json` file.

The converter MUST handle:
- `time.style === "unix"`: store the raw millisecond value in `date::`,
  omit `date-style::` (unix is default).
- `time.style === "free"`: store the value in `date::`, add
  `date-style:: free`.
- `nodePos`: discard (not stored in HMD).
- `fileParent`: discard (derived from source path).
- `isExpanded`: discard (UI-only state).
